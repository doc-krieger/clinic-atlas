# Phase 1: Foundation - Context

**Gathered:** 2026-04-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Docker Compose stack (Postgres 17, FastAPI, Next.js, Ollama, SearXNG), Postgres schema with full-text search and medical synonym dictionary, disk layout for knowledge base files, and trusted source registry loaded from YAML. This phase delivers the infrastructure that all subsequent phases build on.

<decisions>
## Implementation Decisions

### Database Schema
- **D-01:** Unified `notes` table with a `type` column (source_note, topic_note, research_log). Single FTS index covers all note types.
- **D-02:** Separate `raw_sources` table for ingested PDFs/URLs. Tracks file path, URL, parse status, content hash, mime type. Stores extracted text content in a `content` column (enables FTS on raw sources too).
- **D-03:** `note_sources` junction table linking notes to raw sources with granular citation metadata: page_number, section_heading, quote_excerpt. Supports many-to-many relationships.
- **D-04:** Notes table columns: slug, title, content, type, status (draft/approved/archived), tags (text array), version (integer), created_at, updated_at, plus tsvector generated column.
- **D-05:** FTS uses `GENERATED ALWAYS AS` stored tsvector column with weighted indexing — title at weight A, content at weight D. GIN index on the generated column.
- **D-06:** FTS indexing on both `notes` and `raw_sources` tables. Same GENERATED ALWAYS AS pattern on both.
- **D-07:** `research_sessions` table in Postgres with query, timestamps, and JSONB column for chat message log.
- **D-08:** Alembic migrations set up from day one. No create_all — schema changes tracked from the start.
- **D-09:** Version column on notes table (integer counter) for future topic refresh diff support.

### Medical Synonym Dictionary
- **D-10:** Custom Postgres thesaurus dictionary for medical abbreviation expansion. Custom text search configuration that uses the thesaurus.
- **D-11:** Comprehensive seed list (~500+ abbreviations) generated by LLM, covering common clinical abbreviations.
- **D-12:** Ambiguous abbreviations expand to all meanings (e.g., MS → multiple sclerosis AND mitral stenosis). Recall over precision at this scale.
- **D-13:** Include Canadian/British spelling variants (anaemia/anemia, haemoglobin/hemoglobin) in the thesaurus.
- **D-14:** Thesaurus file is a static file in the repo (`config/medical_thesaurus.ths`). Version-controlled, Docker copies into Postgres data directory.
- **D-15:** Thesaurus applied at index time (on tsvector creation). Adding new synonyms requires reindex.

### Disk Layout
- **D-16:** Flat by type: `notes/topics/`, `notes/sources/`, `notes/logs/`. Each note is `{slug}.md` with YAML frontmatter.
- **D-17:** Full frontmatter metadata: title, slug, type, status, tags (list), created, updated, version, sources (list of source slugs).
- **D-18:** Raw files stored in `data/sources/` directory, separate from notes. Content hash prevents duplicates.
- **D-19:** Docker volumes with bind mounts for dev (./data/notes:/data/notes, ./data/sources:/data/sources). Production can use named volumes.
- **D-20:** Title-based slugs: 'Neonatal Jaundice Management' → 'neonatal-jaundice-management'. Numeric suffix for collisions.
- **D-21:** Auto-create directories on FastAPI startup. Zero manual setup — docker compose up just works.
- **D-22:** Atomic disk writes: write to {slug}.tmp, fsync, rename to {slug}.md. Prevents corrupted notes on crash.
- **D-23:** Full reindex endpoint (POST /api/reindex) that rebuilds Postgres from disk files. Validates frontmatter integrity and reports issues (non-blocking).
- **D-24:** Comprehensive health endpoint (GET /api/health) reporting status of all services: Postgres, Ollama, SearXNG, disk volumes.
- **D-25:** Data paths configurable via environment variables with sensible defaults (CLINIC_ATLAS_NOTES_DIR, CLINIC_ATLAS_SOURCES_DIR).

### Source Registry
- **D-26:** Rich metadata per source: name, domain, category (guideline/textbook/journal/formulary), base_url, search_url_pattern, requires_auth (boolean), reliability_tier (authoritative/reference/supplementary), notes (free text).
- **D-27:** YAML organized by category as top-level keys: guidelines, textbooks, journals, formularies. Each contains a list of sources.
- **D-28:** Pre-populated with Canadian clinical sources: CPS, SOGC, Choosing Wisely Canada, UpToDate, DynaMed, CPhA, Alberta Health guidelines, etc.
- **D-29:** Validate and warn at startup — parse with Pydantic model, log warnings for issues, but load valid entries. Startup doesn't fail over one bad entry.
- **D-30:** Fixed path (config/sources.yml) with CLINIC_ATLAS_SOURCES_FILE env var override.
- **D-31:** Reliability tier field for source ranking during research synthesis (authoritative > reference > supplementary).

### LLM Provider Configuration
- **D-32:** Ollama as default local LLM provider in Docker Compose. llama-server (llama.cpp) supported as optional alternative via LLAMA_SERVER_URL env var through litellm's OpenAI-compatible provider prefix.

### Docker Compose
- **D-33:** All 5 services start together — no profiles. Single `docker compose up` runs Postgres, FastAPI, Next.js, Ollama, SearXNG.
- **D-34:** Bind mount source code for hot-reload: ./backend:/app with uvicorn --reload, ./frontend:/app with next dev.
- **D-35:** Docker healthchecks on all services with depends_on condition: service_healthy for proper startup ordering.
- **D-36:** Standard port mappings: Frontend 3000, Backend 8000, Postgres 5432, Ollama 11434, SearXNG 8888.

### FastAPI Project Structure
- **D-37:** Feature-based module layout: app/notes/, app/sources/, app/search/, app/health/ — each with models.py, router.py, service.py.
- **D-38:** Pydantic BaseSettings with .env file for configuration. Single Settings class injected via FastAPI dependency.
- **D-39:** API prefix /api/ without versioning. No /v1/ — single user, no external consumers.

### Frontend Bootstrapping
- **D-40:** Chat layout skeleton from day one: sidebar + main area, /chat route with placeholder components (message-list, message-input). Gives a visual preview of the final product.
- **D-41:** Dark mode default, clinical-clean aesthetic. Muted accent colors, high contrast text, minimal like Linear/Raycast. Light mode toggle available.
- **D-42:** Direct FastAPI connection with CORS (no Next.js proxy). Required for SSE streaming in later phases.
- **D-43:** shadcn/ui initialized with base components: Button, Card, Input, ScrollArea, Separator.

### SearXNG Configuration
- **D-44:** Google + Bing + DuckDuckGo engines configured for redundancy and deduplication.
- **D-45:** JSON API and web UI both enabled. Web UI at localhost:8888 for debugging, JSON API for programmatic access from FastAPI.
- **D-46:** settings.yml checked into repo and mounted into the SearXNG container. Version-controlled and reproducible.

### Testing Infrastructure
- **D-47:** pytest with real Postgres for backend tests. No SQLite substitution — tests hit actual FTS, thesaurus, and migrations. Catches issues mocks would miss.
- **D-48:** Vitest + React Testing Library for frontend component/unit tests. No E2E in Phase 1 — add Playwright later when there's real UI to test.
- **D-49:** Docker Compose test profile: `docker compose --profile test run backend-test` spins up a test Postgres and runs pytest. Frontend tests run locally via `pnpm test`.
- **D-50:** Coverage philosophy: test critical paths (API endpoints, business logic, data integrity), not arbitrary coverage %. Tests should catch regressions.
