## Summary

Phase 1 is a greenfield infrastructure phase delivering five services via Docker Compose (Postgres 17, FastAPI, Next.js, Ollama, SearXNG), a Postgres schema with full-text search using a custom medical thesaurus dictionary, a disk-based knowledge base layout, and a YAML-driven trusted source registry. All decisions are locked from the context session -- no open design questions remain.

The most technically nuanced piece is the PostgreSQL thesaurus dictionary for medical abbreviation expansion. This requires: (1) a `.ths` file mounted into the Postgres container's `tsearch_data` directory, (2) a custom text search configuration created via SQL in an init script, and (3) `GENERATED ALWAYS AS` tsvector columns that reference that custom configuration. The thesaurus must be created before any tables that reference it, making migration ordering critical.

**Primary recommendation:** Structure the work as Docker Compose first, then Postgres schema + thesaurus via Alembic + init scripts, then FastAPI scaffolding with health/reindex endpoints, then source registry, then frontend skeleton. Each layer validates the one below it.

### Pattern 1: PostgreSQL Thesaurus Dictionary for Medical Synonyms

**What:** A custom Postgres text search configuration that uses a thesaurus dictionary to expand medical abbreviations during indexing.

**When to use:** All tsvector columns must reference this custom config instead of the built-in `'english'` config.

**Setup flow (critical ordering):**

1. Mount thesaurus file into Postgres container at `/usr/share/postgresql/17/tsearch_data/medical_thesaurus.ths`
2. Run init SQL that creates the dictionary and configuration
3. Alembic migrations reference the custom config in `GENERATED ALWAYS AS` expressions

**Init SQL (runs via `/docker-entrypoint-initdb.d/`):**
```sql
-- Source: https://www.postgresql.org/docs/current/textsearch-dictionaries.html
-- Create thesaurus dictionary
CREATE TEXT SEARCH DICTIONARY medical_thesaurus (
    TEMPLATE = thesaurus,
    DictFile = medical_thesaurus,
    Dictionary = english_stem
);

-- Create custom text search configuration based on English
CREATE TEXT SEARCH CONFIGURATION medical (COPY = english);

-- Replace word mappings to use thesaurus first, then English stemmer
ALTER TEXT SEARCH CONFIGURATION medical
    ALTER MAPPING FOR asciiword, asciihword, hword_asciipart
    WITH medical_thesaurus, english_stem;
```

**Thesaurus file format (`config/medical_thesaurus.ths`):**
```
# Medical abbreviations - format: phrase : expansion(s)
# Source: https://www.postgresql.org/docs/current/textsearch-dictionaries.html

# Cardiovascular
htn : hypertension
mi : myocardial infarction
chf : congestive heart failure
afib : atrial fibrillation
dvt : deep vein thrombosis

# Ambiguous - expand to all meanings (D-12)
ms : multiple sclerosis mitral stenosis

# Canadian/British spelling variants (D-13)
anaemia : anemia anaemia
haemoglobin : hemoglobin haemoglobin
oedema : edema oedema
oesophagus : esophagus oesophagus
paediatric : pediatric paediatric
```

**Table with weighted tsvector (D-05):**
```sql
-- Source: https://www.postgresql.org/docs/current/textsearch-controls.html
CREATE TABLE notes (
    id SERIAL PRIMARY KEY,
    slug VARCHAR(255) UNIQUE NOT NULL,
    title TEXT NOT NULL,
    content TEXT NOT NULL DEFAULT '',
    type VARCHAR(20) NOT NULL CHECK (type IN ('source_note', 'topic_note', 'research_log')),
    status VARCHAR(20) NOT NULL DEFAULT 'draft' CHECK (status IN ('draft', 'approved', 'archived')),
    tags TEXT[] DEFAULT '{}',
    version INTEGER NOT NULL DEFAULT 1,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    search_vector TSVECTOR GENERATED ALWAYS AS (
        setweight(to_tsvector('medical', coalesce(title, '')), 'A') ||
        setweight(to_tsvector('medical', coalesce(content, '')), 'D')
    ) STORED
);

CREATE INDEX idx_notes_search ON notes USING GIN (search_vector);
```

### Pattern 2: Alembic with SQLModel + Custom FTS

**What:** Alembic manages schema migrations but the thesaurus dictionary and FTS configuration must exist before migrations run.

**Critical ordering:**
1. Docker entrypoint init script (`/docker-entrypoint-initdb.d/01-fts-config.sql`) creates the thesaurus dictionary and `medical` text search configuration
2. Alembic migrations create tables referencing `'medical'` in their generated columns
3. Init scripts run on first `docker compose up`, Alembic runs on every startup

**env.py pattern for SQLModel:**
```python
# Source: Context7 /sqlalchemy/alembic + /websites/sqlmodel_tiangolo
from sqlmodel import SQLModel
from app.config import settings

# Import all models so SQLModel.metadata is populated
```

**Important caveat:** SQLModel does not natively understand tsvector columns or `GENERATED ALWAYS AS`. The generated column must be added via raw SQL in the Alembic migration using `op.execute()`, not via SQLModel model definition. The SQLModel model should exclude the `search_vector` column (it's read-only from the DB). [VERIFIED: Context7 /websites/sqlmodel_tiangolo - FTS not natively supported]

### Pattern 4: Atomic Disk Writes (D-22)

```python
# Source: standard POSIX atomic write pattern
import os
import tempfile

def atomic_write(path: str, content: str) -> None:
    """Write file atomically: write to tmp, fsync, rename."""
    dir_path = os.path.dirname(path)
    fd, tmp_path = tempfile.mkstemp(dir=dir_path, suffix=".tmp")
    try:
        with os.fdopen(fd, "w") as f:
            f.write(content)
            f.flush()
            os.fsync(f.fileno())
        os.rename(tmp_path, path)
    except Exception:
        os.unlink(tmp_path)
        raise
```
[ASSUMED - standard POSIX pattern]

### Anti-Patterns to Avoid

- **Using `'english'` instead of `'medical'` in tsvector expressions:** All `to_tsvector()` calls must use the custom `'medical'` configuration to get thesaurus expansion. Mixing configs means some content won't match medical abbreviation searches.
- **Creating thesaurus dictionary inside Alembic migrations:** The thesaurus file must already be on disk when `CREATE TEXT SEARCH DICTIONARY` runs. It belongs in Docker init, not Alembic.
- **Using `SQLModel.metadata.create_all()` (D-08):** All schema changes go through Alembic. `create_all` would bypass migration history.
- **Proxying SSE through Next.js:** Connect browser directly to FastAPI for SSE (D-42). Next.js buffers SSE responses.
- **Using `ts_lexize` to test the thesaurus:** It treats input as a single token -- use `plainto_tsquery('medical', 'htn')` or `to_tsvector('medical', 'htn')` instead.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Medical synonym expansion | Custom Python synonym lookup | Postgres thesaurus dictionary | Operates at index time, zero application code, handles multi-word phrases, leverages GIN index |

## Common Pitfalls

### Pitfall 1: Thesaurus File Path in Docker
**What goes wrong:** `CREATE TEXT SEARCH DICTIONARY` fails with "could not open file" because the `.ths` file isn't at the expected `$SHAREDIR/tsearch_data/` path.
**Why it happens:** The Postgres 17 Docker image sharedir is `/usr/share/postgresql/17/tsearch_data/`. The path varies by Postgres major version.
**How to avoid:** Mount directly: `./config/medical_thesaurus.ths:/usr/share/postgresql/17/tsearch_data/medical_thesaurus.ths:ro`. Verify with `docker exec <container> pg_config --sharedir`.
**Warning signs:** `ERROR: could not open dictionary file` in Postgres logs.

### Pitfall 2: Init Script vs. Alembic Ordering
**What goes wrong:** Alembic migration references `to_tsvector('medical', ...)` but the `medical` text search configuration doesn't exist yet.
**Why it happens:** Docker init scripts (`/docker-entrypoint-initdb.d/`) only run on first database creation. If the volume already has data, they're skipped. But Alembic runs every startup.
**How to avoid:** Make the init script idempotent (`CREATE ... IF NOT EXISTS` for the text search config). Also add a check in the FastAPI lifespan that verifies the `medical` config exists before running Alembic. Alternatively, create the FTS config in the first Alembic migration itself using `op.execute()`.
**Warning signs:** `ERROR: text search configuration "medical" does not exist`.

### Pitfall 3: SQLModel Cannot Model Generated Columns
**What goes wrong:** Defining `search_vector` as a SQLModel field causes errors on insert/update because Postgres rejects writes to GENERATED columns.
**Why it happens:** SQLModel/SQLAlchemy tries to include all mapped columns in INSERT/UPDATE statements.
**How to avoid:** Do NOT define `search_vector` in the SQLModel class. Create it via raw SQL in Alembic migration (`op.execute()`). For read queries needing FTS, use `text()` or raw SQL: `session.exec(text("SELECT * FROM notes WHERE search_vector @@ plainto_tsquery('medical', :q)"), params={"q": query})`.
**Warning signs:** `ERROR: column "search_vector" can only be updated to DEFAULT`.

### Pitfall 4: SearXNG JSON API Not Enabled by Default
**What goes wrong:** Requests to SearXNG with `?format=json` return 403 Forbidden.
**Why it happens:** SearXNG only enables HTML format by default. JSON must be explicitly enabled in `settings.yml`.
**How to avoid:** Add to `config/searxng/settings.yml`:
```yaml
search:
  formats:
    - html
    - json
```
**Warning signs:** HTTP 403 when hitting `http://searxng:8080/search?q=test&format=json`.

### Pitfall 5: Thesaurus Reindex Requirement (D-15)
**What goes wrong:** New abbreviations added to the thesaurus file don't appear in search results.
**Why it happens:** Thesaurus is applied at index time. Existing tsvector values don't update when the dictionary file changes.
**How to avoid:** The reindex endpoint (D-23) must call `REINDEX INDEX idx_notes_search` or re-generate tsvector values by updating rows. Document this clearly -- adding abbreviations requires a POST to /api/reindex.
**Warning signs:** New abbreviation searches return no results despite matching content existing.

### Pitfall 6: Docker Compose Healthcheck for Ollama
**What goes wrong:** FastAPI starts and immediately tries to connect to Ollama, which is still downloading/loading models.
**Why it happens:** Ollama container is "healthy" (API responds) but no models are loaded yet.
**How to avoid:** The health endpoint (D-24) should report Ollama status separately. FastAPI itself should not block startup on Ollama availability. Backend depends_on only postgres (service_healthy), not Ollama.
**Warning signs:** Ollama connection timeouts on first request.

### Source Registry Pydantic Model + YAML Loading (D-26 through D-31)

```python
# Source: Pydantic v2 docs + project decisions
from enum import Enum
from pydantic import BaseModel
import yaml
import logging

logger = logging.getLogger(__name__)

class ReliabilityTier(str, Enum):
    authoritative = "authoritative"
    reference = "reference"
    supplementary = "supplementary"

class SourceEntry(BaseModel):
    name: str
    domain: str
    category: str  # guideline, textbook, journal, formulary
    base_url: str | None = None
    search_url_pattern: str | None = None
    requires_auth: bool = False
    reliability_tier: ReliabilityTier = ReliabilityTier.reference
    notes: str | None = None

class SourceRegistry(BaseModel):
    guidelines: list[SourceEntry] = []
    textbooks: list[SourceEntry] = []
    journals: list[SourceEntry] = []
    formularies: list[SourceEntry] = []

    @property
    def all_sources(self) -> list[SourceEntry]:
        return self.guidelines + self.textbooks + self.journals + self.formularies

def load_source_registry(path: str) -> SourceRegistry:
    """Load and validate source registry. Warn on bad entries, load valid ones."""
    with open(path) as f:
        raw = yaml.safe_load(f)

    if not raw:
        logger.warning("Source registry file is empty: %s", path)
        return SourceRegistry()

    # Validate each category, skip bad entries
    validated = {}
    for category in ["guidelines", "textbooks", "journals", "formularies"]:
        entries = []
        for item in raw.get(category, []):
            try:
                entries.append(SourceEntry(category=category, **item))
            except Exception as e:
                logger.warning("Skipping invalid source entry in %s: %s - %s", category, item.get("name", "unknown"), e)
        validated[category] = entries

    return SourceRegistry(**validated)
```

### FTS Query with Medical Config

```python
# Source: PostgreSQL docs + SQLModel pattern from Context7
from sqlalchemy import text
from sqlmodel import Session

def search_notes(session: Session, query: str, limit: int = 20) -> list[dict]:
    """Full-text search using medical thesaurus configuration."""
    result = session.exec(
        text("""
            SELECT id, slug, title, type, status,
                   ts_rank(search_vector, query) AS rank
            FROM notes,
                 plainto_tsquery('medical', :q) AS query
            WHERE search_vector @@ query
            ORDER BY rank DESC
            LIMIT :limit
        """),
        params={"q": query, "limit": limit}
    )
    return [dict(row._mapping) for row in result]
```

## Open Questions

1. **Thesaurus in Alembic vs. init script**
   - What we know: The thesaurus dictionary + FTS config must exist before tables reference them. Docker init scripts only run on first volume creation.
   - What's unclear: Whether to put FTS config creation in the init script OR the first Alembic migration.
   - Recommendation: Use both -- init script for fresh installs, first Alembic migration with `IF NOT EXISTS` for safety. The Alembic migration is the authoritative source; the init script is a shortcut for clean environments.

2. **SQLModel generated column exclusion**
   - What we know: SQLModel can't model GENERATED columns. The column must be created via raw SQL.
   - What's unclear: Best pattern for reading tsvector results back in SQLModel queries.
   - Recommendation: Use `session.exec(text(...))` for FTS queries. Don't try to map tsvector to a SQLModel field.

3. **Thesaurus reindex mechanism**
   - What we know: Changing the thesaurus file requires reindexing. D-23 specifies a reindex endpoint.
   - What's unclear: Whether `REINDEX INDEX` is sufficient or whether rows need UPDATE to regenerate the generated column.
   - Recommendation: `REINDEX INDEX` alone is NOT sufficient for generated columns -- the stored tsvector values won't change. The reindex endpoint must UPDATE each row to trigger regeneration: `UPDATE notes SET updated_at = NOW()` (any column change triggers generated column recalculation). Actually, for `GENERATED ALWAYS AS STORED` columns, only changes to the source columns (title, content) trigger recalculation. The reindex endpoint should: (a) drop and recreate the generated column definition, or (b) use `ALTER TABLE ... ALTER COLUMN search_vector SET GENERATED ALWAYS AS (...)` to force recalculation. Research this further during implementation.
